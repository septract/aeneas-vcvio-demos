# Trust Ledger

*Last updated: 2026-05-30.*

This file tracks **where the demos introduce trust** — the surfaces a reader must believe in
order to believe the headline theorems mean what we say they mean. It exists because the
machine-checked soundness gate (`make verify`) and the kernel check *do not* cover the most
important surface: whether a **definition** captures the notion we intend. A perfectly sound proof
of the wrong definition is worthless, and the kernel cannot tell the difference.

## What `make verify` does and does **not** cover

- **Covers:** every headline theorem (listed in `demos/lean/Audit.lean`) depends transitively on
  *only* `[propext, Classical.choice, Quot.sound]` — no `sorry`/`sorryAx`, no `native_decide`
  (`Lean.ofReduceBool`), no custom `axiom`. This is a genuine *no-cheating* gate on the proofs.
- **Does NOT cover:** definitional adequacy (does our game/scheme model the real notion?), the
  faithfulness of Aeneas extraction, or the named hardness assumptions. Those are tracked here.

## Trust classes

Each surface below is one of:

- **(T) Trusted-reused** — a VCVio library definition we use unchanged. Trust is *socialized*:
  it lives in a shared, reviewed, published library, not invented here.
- **(C) Defined-here, cross-checked** — defined in this repo. Trust is *local*; we contain it with
  behavioural cross-check lemmas and/or by anchoring to a (T) definition. Each (C) item is a place
  a non-cryptographer supervisor must check the *statement*, not just the proof.
- **(A) Irreducible assumption** — a named hardness/primitive assumption we deliberately bottom out
  on (the leaves of the reduction tree). Not proved; assumed, with precedent.
- **(X) External toolchain trust** — trusted software outside the Lean proof (the extractor).

| Surface | Class | Notes / precedent |

The remaining layers are the per-demo security definitions, constructions, and assumptions.

### The single (C) security definition: `SUF_CMA_Exp`

This is the only place in the repo where we define a security *game* not inherited from VCVio, so
it gets explicit justification:

1. **Precedent.** Strong unforgeability (freshness on the `(message, tag)` *pair* rather than the
   message) is the standard textbook notion — Bellare–Namprempre, *Authenticated Encryption*
   (ASIACRYPT 2000); Goldreich, *Foundations of Cryptography* Vol. 2. It is *not* a novel model.
2. **Structural minimality, machine-checked.** `SUF_CMA_Exp` is VCVio's trusted `UF_CMA_Exp` body
   *verbatim* with a single swapped predicate (`!log.wasQueried msg → !wasQueriedPair log msg τ`).
   `UF_CMA_Exp_eq_core` *proves* the shared experiment core equals VCVio's `UF_CMA_Exp`, and
   `SUF_CMA_Exp_eq_core` shows the SUF game reuses that same core. The only semantic novelty is one
   predicate.
3. **Behavioural pins.** `suf_gate_iff` (the gate accepts *exactly* a fresh pair carrying the
   correct tag — replays rejected, genuine forgeries counted) and `sufAdv_eq_ufAdv` (strong and
   plain advantages are *equal* for this canonical MAC, proved in **both** directions — the
   MAC-agnostic `≥` via `wasQueried_of_wasQueriedPair` confirms the game is genuinely *stronger*,
   not accidentally weaker or vacuous).

A non-cryptographer supervisor can validate (2) and (3) mechanically; (1) is the precedent that
makes the notion the right one.

## Deferred — require the team's cryptographers (NOT in the trusted base; no clean in-tool precedent)

These were considered and **intentionally not built**, because they would require defining novel
security *models* with no trusted VCVio reference — a cryptographer's judgement, not something to
improvise. They are recorded here so the boundary is explicit.

| Candidate | Why it needs an expert |
|---|---|
| PQXDH **KEM⊕DH robust combiner** (key pseudorandom under KEM-IND-CCA *or* DH) | a specialized security model with no trusted reference; needs the combiner game defined/audited by a cryptographer (cf. PQXDH CryptoVerif analysis; Bindel et al. hybrid-KEX; Giacon–Heuer–Poettering KEM combiners) |
| **Encrypt-then-MAC AEAD** | VCVio has no computational IND-CPA / INT-CTXT / AE game for *symmetric* encryption (only perfect secrecy); would require ~3 new game definitions — textbook, but enough invention to want a cryptographer to bless them first |
| **HMAC-is-a-PRF** | deep, novel infrastructure (Merkle–Damgård cascade, dual-PRF); discharges Demo 4's (A) assumption but is paper-sized |
| **Multi-user UF-CMA** | a new (multi-key) game |

**The supervisability boundary ≈ the precedent boundary.** Work that reuses a trusted VCVio game,
mirrors a textbook notion anchored to one, or reuses existing infra is supervisable by a
formal-methods-literate non-cryptographer. Defining a novel security model is not.

## Maintaining this ledger

When adding a demo or theorem: add its security definitions, constructions, and assumptions here,
classified (T)/(C)/(A)/(X). Any new **(C) security definition** must carry a precedent citation and
behavioural cross-checks (the `SUF_CMA_Exp` pattern above), or be moved to the deferred section.
Date-stamp significant updates at the top.
