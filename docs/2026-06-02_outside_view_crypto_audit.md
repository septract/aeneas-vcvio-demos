# Outside-View Cryptographic Audit — libsignal-theory

## Verdict

This is a real, kernel-checked, and — by the standards of formal-crypto artifacts — unusually honest piece of work. The reused VCVio games I and the panel could locate are the genuine textbook notions (Shannon perfect secrecy over all priors, adaptive multi-query EUF-CMA, the standard PRG/PRF distinguishing games, the textbook 2-KEM+1-DEM hybrid), the OTP earns its ε=0 over the full key/message space, the one hand-defined game (SUF-CMA) is pinned down by genuine two-direction cross-checks, and the hybrid/advantage accounting is sound (correct telescoping, load-bearing poly-length premise, the 1/|Tag| term carried not dropped). Against that strong baseline, the **single material defect that survives independent skeptical cross-checking** is a localized but high-stakes prose over-claim in the PQXDH node: it ties the Bhargavan et al. (USENIX'24) re-encapsulation attack to the byte-layout glue these nodes model, when — as I verified against both the modeled code and the paper itself — the attack-relevant binding (the KEM public key PQSPK into the AEAD associated data) is exactly what is *not* present in the modeled secret input or AD. Every other "major" the panel raised is a technically-true observation about idealized assumptions whose over-claim/scope severity is **refuted by the artifact's own prominent disclosures**, and so drops to minor/note. **Overall verdict: sound-but-overclaimed-in-places.** I would accept this as a faithful methodology demonstration with a real soundness gate, conditioned on de-claiming the BJKS attribution and tightening a handful of point-of-claim qualifiers.

## What the artifact actually delivers (calibrated, grounded)

Judged independently of the marketing, the artifact delivers four genuinely solid things and one over-stated thing:

1. **Faithful instantiation of standard symmetric-crypto security notions in VCVio, machine-checked.** `perfectSecrecyAt` is real Shannon independence quantified over *all* message priors (`SymmEncAlg.lean:130-205`), and the OTP instantiates it over the full `BitVec 64` key and message space (`OneTimePad.lean:24-56`) — ε=0 is earned, not cheap. The MAC game is genuine adaptive multi-query EUF-CMA with message-freshness logging (`MacAlg.lean:37-91`), backed by a proper lazy-random-oracle PRF game (`PRF.lean:44-89`).

2. **A sound n-step hybrid for the KDF ratchet.** The telescoping is correct (glue lemma `gen_ideal_eq_next`, boundary hybrids, `Finset.sum_range_sub'` then `abs_sum_le_sum_abs`; `Generic.lean:127-260`), each per-step reduction is a *legitimate single-instance* PRG distinguisher (consumes the challenge block once, computes G itself elsewhere; `Generic.lean:62-74`), and the poly-length premise is genuinely load-bearing (`negligible_polynomial_mul`; drop it and `Generic.lean:260` does not close). The forward-secrecy result is the genuinely stronger *joint* (keystream, surviving chain key)-vs-uniform statement (`ForwardSecrecy.lean:31-32,137-158`), not re-notated keystream pseudorandomness.

3. **One new, non-vacuous, correctly-stronger defined-here game.** `SUF_CMA_Exp` is VCVio's `UF_CMA_Exp` body with a single swapped gate predicate (factored through a shared `macGameCore`, machine-checked by `*_eq_core`; `SufCma.lean:536-566`), freshness is on the correct `(msg,τ)` *pair* field (`SufCma.lean:407-421`), the gate accepts exactly fresh-and-correct (`suf_gate_iff`, `SufCma.lean:440-445`), and `sufAdv_eq_ufAdv` is proved in *both* directions — the UF≤SUF direction uses no determinism (`SufCma.lean:625-644`), confirming SUF is genuinely the stronger notion that *collapses* for this canonical MAC rather than an accidentally weaker game.

4. **A real soundness gate matching the headline list.** `audit.sh` blocks `sorryAx`/`ofReduceBool`/`native_decide` and the reflection axioms (`_native`/`bv_decide.ax`), and the expected count reconciles with the `#print axioms` lines in `Audit.lean` (per the positioning reviewer). (Per the brief, kernel soundness is stipulated, not re-litigated.)

5. **Over-stated:** the protocol nodes (PQXDH/SPQR) prove *byte-layout functional correctness + totality* of hand-written analog Rust, with **no security property**, and the framing periodically reads as if the byte layouts carry attack-relevant security content (see findings).

## Material findings (ranked)

### M1 — [MAJOR] BJKS re-encapsulation attribution over-claim (CONFIRMED by my own spot-check)
- **Grounding (verbatim):** `KeySchedule.lean:11` ("the error-dense layer the Bhargavan et al. (USENIX'24) re-encapsulation attack lived in"), `:23` and `:344` ("the segment ordering is the part the Bhargavan et al. re-encapsulation attack turned on" / "BJKS-attack-relevant part"), `README.md:229` (table cell) and `README.md:257-258` ("the published PQXDH attack was a domain-separation bug in exactly this glue").
- **Skeptic verdict:** confirmed.
- **My own spot-check (I read the code and paper directly):** The modeled secret input is `[0xFF;32]‖DH1‖DH2‖DH3‖SS` (`pqxdh_secret_input_spec`, `KeySchedule.lean:346-352`), where `SS` is the **KEM shared secret**, not the public key — `demos/rust/pqxdh/pqxdh.rs:55` states verbatim `SS = ML-KEM shared secret encapsulated to the Kyber prekey`. The modeled AD is `EncodeEC(IK_A)‖EncodeEC(IK_B)` — identity keys only (`pqxdh.rs:203`, `associated_data_spec`). The Bhargavan paper's re-encapsulation fix, which I extracted directly, reads: *"the PQSPK should be included inside the Authenticated Data of the AEAD, or the KEM should ... bind together the secret and the public key"* (`deps/papers/usenixsecurity24-bhargavan.pdf`, extracted text line 1128; the attack figure shows `AD = IKApk ‖ IKBpk`, lines 977/98). The attack-relevant binding (PQSPK/CT into AD or KDF) is therefore **absent from exactly the glue these nodes model**, and the separate public-key-confusion attack concerns a KEM-key encoding the codec (DJB 0x05 only) does not model either. The prose attributes the most consequential named attack to a byte layout that is not where the attack lived, with no disclaimer at point of claim. The reviewer's `README:558-561` citation is stale (the file is 367 lines; the real loci are `:229` and `:257-258`), but the substance is exact.
- **Recommendation:** Re-scope to "the same *class* of domain-separation/byte-layout glue" without asserting it is the specific construction the re-encapsulation attack exploited; or model the PQSPK/CT-in-AD binding (the actual locus) before making the attack-specific claim. This is the one change I would *require* before a top-venue submission.

### M2 — [MINOR, downgraded from major] "Poly-query" class admits exponential-time adversaries
- **Grounding:** `QueryBound.lean:67-69,676-677` (`pure x` satisfies any bound including 0; budget decrements only at query nodes), `Cost.lean:109-127`, `README.md:126-135`.
- **Skeptic verdict:** partial — technical claims confirmed, severity refuted.
- **My own spot-check:** `IsTotalQueryBound` does count only uniform-sampling oracle queries, and an adversary doing brute-force search in pure code has bound 0; so the class literally contains exponential-time adversaries and is *not* a PPT class. But this is **disclosed prominently and accurately**: `README.md:346-354` is a dedicated "honest open items" section whose first bullet is "Cost measure is query count, not running time," and `README.md:352-353` states verbatim "The other demos' base advantage (`prgAdvantage`) still quantifies over *all* adversaries — only the ratchet adds the cost-bounded statement." `Cost.lean:2-5` says the same in-source. The artifact does not pass query-count off as PPT.
- **Recommendation:** Phrase the headline as "secure against *query-bounded* distinguishers," and keep the query-vs-time caveat adjacent to every "poly-query" use. No soundness impact.

### M3 — [MINOR, downgraded from major] Stream/KEM/DEM asymptotic antecedents quantify over unbounded adversaries
- **Grounding:** `PRG.lean:43,59-63` (`PRGAdversary R := R → ProbComp Bool`, no cost field), `Word.lean:95-104`, `Composition.lean:326-344`, `Negligible.lean:27`.
- **Skeptic verdict:** partial — mechanics confirmed, scope-honesty framing refuted.
- **My own spot-check:** It is true that an unqualified "secure PRG" over this definition is unsatisfiable against unbounded distinguishers (range-membership gives advantage 1), so these theorems are sound *implications* with an idealized premise. This is the standard provable-security convention (a reduction's value is the implication), and the artifact labels the theorems "(conditional)" (`README.md:97,102,195`) and tags the efficiency observation "*informal*" in-source (`Word.lean:75-78`). I confirmed `README.md:352-353` exists and says exactly this. Honest as documented.
- **Recommendation:** Optionally promote `ratchet_secure_against_polyQuery`'s class-relative form as the model for the other demos; otherwise no change.

### M4 — [MINOR] Mirror-faithfulness is the dominant uncheckable trust
- **Grounding:** `TRUST.md:155-158` (labels analog-Rust = deployed-libsignal machine-uncheckable), per-node table cells in `README.md:227-247` carry the correspondence claim without the qualifier at point of use; XREF tags are per-line (`[type-only]`/one `[domain-restricted]`).
- **Skeptic verdict:** not separately adjudicated; cross-corroborated by two reviewers.
- **My note:** The Aeneas guarantee certifies analog-Rust→Lean, *not* analog-Rust→deployed-libsignal. The latter — exactly where BJKS-class risk lives — is an agent-authored free-text claim nothing mechanical checks. TRUST.md is honest about this; the README table cells are where it can be over-read.
- **Recommendation:** State the trust chain explicitly at the README table (kernel certifies P(extracted Lean); Aeneas certifies extracted Lean = analog Rust; analog Rust = libsignal is unchecked).

### M5 — [MINOR] SUF-CMA is the single-verification-attempt variant, not flagged
- **Grounding:** `MacAlg.lean:64-65` (adversary has only `M →ₒ T`, one final forgery; no verify oracle), `SufCma.lean:256-308` (single `1/|Tag|` term).
- **My note:** Standard textbook simplification, faithfully reflected by the types; the `1/|Tag|` tightness is correct *per attempt*. The multi-verification-query variant (with a `q_v/|Tag|` factor) is out of scope and not called out.
- **Recommendation:** One sentence in the SUF docstring/TRUST noting this is the no-verify-oracle variant.

### M6 — [NOTE] "Genuine arithmetic"/"meaty" framing for ChaCha/SHA
- **Grounding:** `Chacha.lean:131-135` (`chacha_ratchet_advantage_le_sum := ratchet_advantage_le_sum chachaPure n A`, verbatim delegation to a generic theorem over any `G:Key→Blk64`), `README.md:80,117-125`.
- **Skeptic verdict:** partial — technically confirmed, over-claim severity refuted.
- **My own spot-check:** Confirmed the 20 ARX rounds are cryptographically inert in the theorem (any total `Key→Blk64` satisfies it identically; only totality is proved of the arithmetic). But `README.md:80` states verbatim "value adequacy = totality of the ARX code; security = the generic hybrid bound at `G := chachaPure`," and `:124-125` names "ChaCha20 keyed by the chain key is a PRG" as the assumption. The text separates the two; it does not credit the arithmetic with security. Not an over-claim as written.
- **Recommendation:** None required; the existing separation is adequate.

### M7 — [NOTE] FS-implies-keystream "stronger than" is prose-only, never machine-checked
- **Grounding:** `ForwardSecrecy.lean:10-11`, `README.md:143`; no Lean lemma links the two headlines.
- **My note:** In an artifact whose selling point is "machine-checked, not trusting the agent," an asserted implication between two headline theorems is an unverified claim. The relationship is standard and true, but cheap to formalize via a `Prod.fst`-postcomposed adversary.
- **Recommendation:** Add the one-line corollary, or soften to "mathematically stronger (not formally derived here)."

### M8 — [NOTE] Stale under-claim comment in KeySchedule.lean

## Definitional adequacy (reused VCVio games + the one defined-here SUF-CMA game)

The reused games are adequate and faithfully instantiated. The genuine adequacy *nuances* (all minor/note, all about scope not soundness): the DEM/KEM/PKE IND-CPA games are **one-time** (no encryption oracle; `DataEncapMech.lean:53-87`, `AsymmEncAlg/INDCPA/OneTime.lean:37-55`) — the prose says "one-time" but the KEM/DEM *advantage identifiers* drop "OneTime," which a skimmer could over-read. The defined-here `SUF_CMA_Exp` is the one new trust boundary; its adequacy rests on `suf_gate_iff` and the two-direction `sufAdv_eq_ufAdv`, which I and two reviewers independently confirmed are genuine (the membership predicate is on the exact `⟨msg,τ⟩` sigma-pair; the ≥ direction is MAC-agnostic). The "SUF buys nothing over UF for this canonical MAC" caveat is real but openly disclosed (`SufCma.lean:393-403`, `TRUST.md:105-109`) — honest, not hidden. Adequate as built.

## Reduction / hybrid / cost-model soundness

The reductions and advantage accounting are sound — stronger than the typical formal-crypto artifact. Stream-cipher results are genuine tight advantage *equalities* (`Word.lean:61-73`) via XOR-as-permutation, not inequalities hiding slack. The ratchet hybrid is a correct telescoping with legitimate single-instance reductions whose query bound is a *proven theorem* (`Cost.lean:50-82`), and the cost-relative ratchet result *derives* the per-hop bound from a class-relative PRG assumption rather than assuming it (`Cost.lean:109-128`) — the more honest formulation. The KEM/DEM factor-of-2 bias-to-distinguishing conversion is correctly applied (`Composition.lean:163-244`). The cost *model* itself is query-count, not PPT (M2): sound as a measure of the *reduction's overhead*, but the word "efficient" must not be read as wall-clock/circuit time — and the artifact says so.

## Extraction→security bridge & hardness assumptions

Where a security claim exists (Demo 3 ratchet), the bridge is soundly wired: the game's oracle genuinely runs the extracted function (`prgRealExp (genBlockPRG chachaPure)` feeds a uniform seed through extracted ChaCha; `Generic.lean:47-55`, `Chacha.lean:131-135`), and the load-bearing pure functions are pinned to the extracted code by universally-quantified totality/value lemmas (`chachaPure_spec`, `ratchet_split_eq`, `enc_spec`), so a deviation surfaces as a proof failure, not silent unsoundness — the "spec lemma silently becomes the definition" trap is avoided. Two honest caveats the panel surfaced: (a) the assumption is really "the ChaCha20 *core* with fixed nonce/counter, over a uniform 256-bit seed, is a one-block PRG" — a *derived* corollary of the standard ChaCha-core-is-a-PRF assumption, presented as "the standard, named" one (`README.md:124-125`); and (b) only *totality* of the arithmetic is proved — there is **no RFC-8439/HMAC test vector or reference-correctness lemma**, so the labels "ChaCha20"/"HMAC-SHA256" rest on trusting the source code is correct ARX, not on a machine-checked correspondence. For the *conditional* security statement this is sound (a buggy primitive just makes the PRG assumption false and the theorem vacuously honest), but "genuine cryptographic arithmetic" should not be read as "verified to be ChaCha20." In Demos 2/4/5 the extracted Rust is only the non-cryptographic glue (XOR, tag-comparison) with the hardness on abstract `prg`/`F_k`/KEM.

## PQXDH/SPQR nodes: framing honesty & spec correctness

**Spec correctness is strong and verifiable.** Every byte-layout statement the panel cross-checked against the pinned upstream (libsignal `5441a83`, SPQR `f2589fe`) is byte-exact: the `0xFF^32‖DH1‖DH2‖DH3[‖DH4]‖SS` secret input, the 0/32/64 derive split, the 0x05-tagged EC codec round-trip, and on the SPQR side the `salt=[0;32]`/`IKM=root_key‖k` swap (openly documented as a follow-the-code decision), the KDF split, and the `label‖epoch‖{hdr,ct1‖ct2(1088)}` MAC inputs. Divergences are honestly tagged. **Framing is structurally honest** — `TRUST.md` and `Audit.lean` gate only `*_total`/`*_spec`/`*_roundtrip` (no security headline), and the internal RFC is emphatically marked not-public strawman. **The one framing defect is M1** (the BJKS attribution), plus three notes: `associated_data` reconstructs an AD buffer upstream never materializes (it feeds HMAC incrementally; `pqxdh.rs:188-214`) attributed to "spec §3.3"; the present-tense "the SCKA game quantifies over this construction" presupposes a game that does not exist and (per the RFC) has an unresolved deployment-fidelity gap; and `decode_ec` is `[domain-restricted]` to exactly `[u8;33]` (stricter than upstream's trailing-tolerant parser) at a security-relevant codec.

## Positioning vs state of the art & the strongest over-claim critique

The novelty claim — an Aeneas→VCVio bridge delivering the *missing machine-checked computational layer* with a clean axiom gate, correctly bounded against EasyCrypt/CryptoVerif/F\*-hax — is defensible and correctly scoped; the deferred-games RFC is genuinely careful (it flags that no off-the-shelf AKE model captures Signal, the LEAK+r-BIND vs MAL-BIND distinction where ML-KEM fails MAL, and the GAP-4 handshake→channel composition that no paper provides). **The strongest fair over-claim critique** is the systematic mismatch between "security-bearing" and "Rust-grounded": for Demos 1-3 all cryptographic content lives in hand-authored VCVio model code under an abstract PRG assumption, with the extracted Rust a trivial total leaf; the link is genuinely tighter only for Demos 4-5 where extracted MAC-verify / XOR-DEM enter the win condition. This is *disclosed* (README:80 separates value-adequacy from security), so it is a positioning emphasis issue, not dishonesty — but a PC reader should not infer "extracted crypto carries the security" for Demos 1-3.

- **VCVio internals** — the reused composition/bias/triangle lemmas (`ind_cpa_one_time_bias_advantage_compose_with_dem_le`, `boolDistAdvantage_triangle`) and the `ProbCompRuntime` monad-morphism coherence were read at the statement level and treated as trusted (T); not re-derived.
- **Aeneas/Charon extraction faithfulness** (`Demos/Extracted/*` = the `.rs`) — the (X) extractor boundary, gitignored/build-generated and not present; analog-`.rs` was checked against upstream but the extracted Lean the theorems quantify over was not diffed line-by-line.
- **analog-Rust = deployed libsignal/SPQR** — the (C) machine-uncheckable surface (M4); upstream file:line ranges were spot-checked by reviewers but I did not re-open every cited span.
- **The PQXDH spec PDF** — not present under `deps/papers/`; the cited spec §2.1/§3.3 section numbers and "AD is exactly EncodeEC(IK_A)‖EncodeEC(IK_B) with no framing" could not be confirmed against the primary spec (only against libsignal source). Whether the modeled AD matches *post-fix PQXDHv2* (PQSPK-in-AD) or *pre-fix* PQXDH could not be confirmed.
- **Bhargavan paper figures** — read via `pdftotext` grep, not full figure rendering; my M1 confirmation rests on the extracted attack/fix text (esp. line 1128 and the `AD = IKApk‖IKBpk` figure lines), which is sufficient for the attribution claim but not a full read of Figure 3.
- **RFC primary citations** (FG Fig 9, TR Thm 4.2, Schmieg 2024/523, SPQR seed-extraction) — assessed for internal consistency only; primary PDFs partly inaccessible.

## Bottom line for the maintainers

This is a sound, honestly-scoped artifact whose formal content is real and whose disclosures pre-empt most hostile over-claim attacks. **Fix exactly one thing before any external submission: de-claim the BJKS re-encapsulation attribution (M1)** — re-scope it to "the same class of domain-separation/byte-layout glue," because the attack-relevant PQSPK-in-AD binding is verifiably absent from the modeled secret input and AD. Then tidy the cheap items: add the one-line FS⇒keystream corollary (M7), fix the stale "future work" comment (M8), add the SUF no-verify-oracle sentence (M5), state the analog-Rust≠deployed-libsignal trust chain at the README table (M4), and re-label "ChaCha20 keyed by the chain key is a PRG" as a derived corollary noting the seed-uniformity modeling step. None of the cost-model/PRG-unbounded findings (M2, M3) require code changes — they are already disclosed; just keep the caveat adjacent to every "secure"/"poly-query" headline so a standalone restatement cannot drop it. With M1 addressed, I would accept this as a faithful methodology demonstration with a genuine soundness gate.
